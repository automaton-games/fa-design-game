# レビューを受けてマージする

> [← 目次に戻る](README.md) | [前: 08 Pull Requestを作る](08-create-pull-request.md) | [次: 10 Projectのステータスを動かす](10-update-project-status.md)

## このページで扱うこと

PRのレビューとマージについて説明します。
自分のPRがレビューされる場合と、ほかの人のPRをレビューする場合の両方を扱い、最後にマージまで進めます。

## なぜレビューをするのか

レビューは、変更がmainに入る前に、ほかのメンバーの目で問題がないか確かめるためのものです。
mainは全員の作業の前提になるため、そこに問題が混入すると影響が広がります。

レビューで指摘が出れば、mainに入る前に修正できます。
そのため、レビューを通ってからマージします。

## レビューする

チームのメンバーは、自分のPRを出すだけでなく、ほかの人のPRもレビューします。
自分がレビューを頼まれたら、次のように進めます。

まず、PRの「概要」「関連Issue」「変更内容」を読んで、何のために何を変えた変更かを把握します。
次に `Files changed` を開き、差分を順に見ます。

<!-- スクリーンショット: Files changed と Review changes の画面 -->

気になった点があれば、該当の行を選んでコメントを書きます。
「どこを、どう直してほしいか、なぜ」を具体的に書くと、相手が対応しやすくなります。

確認が終わったら、右上の `Review changes` からレビューを送ります。
状況に応じて、次のいずれかを選びます。

- **Comment**：質問や気づき（修正の強制はない）
- **Request changes**：修正してほしいことがある
- **Approve**：問題がない。マージしてよい

`Approve` が出ると、PRはマージできる状態になります。

## レビューを待つ

PRを作ったら、レビュアーがコメントを書くのを待ちます。
自分からレビューを依頼するコメントを添えると、レビューが早く始まりやすいです。

## 指摘に対応する

レビューで指摘を受けたら、同じブランチで修正します。
修正は、これまでと同じ手順でコミットしてpushします。
詳しくは [06 コミットする](06-commit.md) を見てください。

pushすると、変更が同じPRに自動で追加されます。
新しくPRを作り直す必要はありません。

指摘への返事はPRのコメント欄に書きます。
対応した項目には `Resolved` などを付けて、見る人に分かりやすくします。

## コンフリクトを直す

pushしたとき、あるいはPRの画面で「コンフリクト（競合）がある」と表示されることがあります。
これは、ほかの人の変更と自分の変更が同じ場所で重なった状態です。

コンフリクトの解決は、このページの範囲を超えます。
起きたときは、PRにコメントして相談してください。

## マージする

レビューが通ったら、PRをmainにマージします。
マージとは、変更をmainに取り込む操作です。

マージの操作は、原則としてレビューを依頼した人、または管理者が行います。
PR本文に `Closes #Issue番号` を書いていれば、マージと同時にIssueも閉じます。

## ブランチを削除する

マージが終わったら、使わなくなった作業ブランチを削除します。
GitHubのPR画面に `Delete branch` ボタンが出るので、これを押します。
手元のブランチも不要なら消します。

```bash
git switch main
git pull
git branch -d <ブランチ名>
```
