## GitとGitHub

**Git（ギット）** は、ファイルの変更履歴を記録して管理する仕組みです。
コードを昔の状態に戻したり、複数人の変更をまとめたりできます。

**GitHub（ギットハブ）** は、Gitで管理したコードをインターネット上で共有し、チームで共同作業するためのサービスです。
このリポジトリもGitHub上にあります。

## リポジトリとブランチ

**リポジトリ** は、ひとつのプロジェクトのファイルと変更履歴をまとめた箱です。
このプロジェクトのコードも、リポジトリに入っています。

リポジトリには、みんなが基準にする **main（メイン）ブランチ** があります。
完成した変更が順に蓄積され、いつでも動く状態を保つのがmainの役割です。

**ブランチ** は、mainから分岐させた独立した作業場所です。
各自がブランチで作業すれば、ほかの人の作業とぶつからずに変更を進められます。

## 開発の全体像

このプロジェクトの開発は、おおむね次の流れで進みます。

1. **Issue** で、やることをひとまとまりにする
2. 担当者が決まったら、mainからブランチを作る
3. ブランチで変更し、**コミット**で変更を記録する
4. **push** で、自分の変更をGitHub上のリポジトリに送る
5. **Pull Request** で、mainに取り込むよう提案する
6. **レビュー**を受けて、問題がなければmainに**マージ**する
7. **GitHub Projects** で進捗を共有する

この進め方では、mainに直接変更を入れず、ブランチで作業してからレビューを通します。
未完成の変更がmainに入ると全員の前提が崩れるためです。

## 用語集

この手順書で使う主な用語をまとめます。

- **リポジトリ**：プロジェクトのファイルと変更履歴をまとめた箱。
- **main**：みんなが基準にするブランチ。完成した変更が蓄積する。
- **ブランチ**：mainから分岐させた独立した作業場所。
- **コミット**：変更をひとまとまりとして記録すること。
- **push**：手元のコミットをGitHub上のリポジトリに送ること。
- **Issue**：やることをひとまとまりにしたカード。作業の単位。
- **Pull Request（PR）**：自分のブランチの変更をmainに取り込む提案。
- **レビュー**：ほかのメンバーが変更内容を確認すること。
- **マージ**：変更をmainに取り込むこと。
- **コンフリクト**：複数人の変更が同じ場所で重なり、自動でまとめられない状態。
- **GitHub Projects**：IssueやPRを並べて進捗を見える化するボード。
