# レビューを受けてマージする

> [← 目次に戻る](README.md) | [前: 07 Pull Requestを作る](07-create-pull-request.md) | [次: 09 Projectのステータスを動かす](09-update-project-status.md)

## このページで扱うこと

PRを作ったあとは、レビューを受け、指摘に対応して、mainにマージします。
このページでは、その一連の流れを説明します。

## レビューを待つ

PRを作ったら、レビュアーがコメントを書くのを待ちます。
自分からレビューを依頼するコメントを添えると、レビューが早く始まりやすいです。

## 指摘に対応する

レビューで指摘を受けたら、同じブランチで修正します。
修正は、これまでと同じ手順でコミットしてpushします。
詳しくは [06 変更をコミットしてpushする](06-commit-and-push.md) を見てください。

pushすると、変更が同じPRに自動で追加されます。
新しくPRを作り直す必要はありません。

指摘への返事はPRのコメント欄に書きます。
対応した項目には `Resolved` などを付けて、見る人に分かりやすくします。

## コンフリクトを直す

pushしたとき、あるいはPRの画面で「コンフリクト（競合）がある」と表示されることがあります。
これは、ほかの人の変更と自分の変更が同じ場所で重なった状態です。

コンフリクトの解決は、このページの範囲を超えます。
起きたときは、PRにコメントして相談してください。

## マージする

レビューが通ったら、PRをmainにマージします。
マージとは、変更をmainに取り込む操作です。

マージの操作は、原則としてレビューを依頼した人、または管理者が行います。
PR本文に `Closes #Issue番号` を書いていれば、マージと同時にIssueも閉じます。

## ブランチを削除する

マージが終わったら、使わなくなった作業ブランチを削除します。
GitHubのPR画面に `Delete branch` ボタンが出るので、これを押します。
手元のブランチも不要なら消します。

```bash
git switch main
git pull
git branch -d <ブランチ名>
```
